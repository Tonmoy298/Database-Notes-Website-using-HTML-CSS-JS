03 / Beginner
DDL — CREATE, ALTER, DROP
Data Definition Language — database এর structure তৈরি এবং পরিবর্তন করার জন্য।
🏗️CREATE TABLE
syntax + example
-- Database তৈরি CREATE DATABASE ecommerce; USE ecommerce; -- Table তৈরি CREATE TABLE users ( id INT PRIMARY KEY AUTO_INCREMENT, name VARCHAR(100) NOT NULL, email VARCHAR(150) UNIQUE NOT NULL, age TINYINT DEFAULT 18, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP );
products table
CREATE TABLE products ( id INT PRIMARY KEY AUTO_INCREMENT, name VARCHAR(200) NOT NULL, price DECIMAL(10,2) NOT NULL, stock INT DEFAULT 0, category_id INT, FOREIGN KEY (category_id) REFERENCES categories(id) );
✏️ALTER TABLE
-- নতুন column যোগ ALTER TABLE users ADD COLUMN phone VARCHAR(20); -- Column এর type পরিবর্তন ALTER TABLE users MODIFY COLUMN phone VARCHAR(15) NOT NULL; -- Column rename ALTER TABLE users RENAME COLUMN phone TO mobile; -- Column delete ALTER TABLE users DROP COLUMN mobile; -- Primary Key যোগ ALTER TABLE users ADD PRIMARY KEY(id); -- Index যোগ ALTER TABLE users ADD INDEX idx_email (email);
🗑️DROP vs TRUNCATE vs DELETE
Commandকী করেRollback?Structure?
DROP TABLETable সহ সব মুছে❌মুছে যায়
TRUNCATEসব data মুছে, structure রাখে❌থাকে
DELETECondition দিয়ে data মুছে✅থাকে
DROP TABLE users; -- table ই নেই TRUNCATE TABLE users; -- data নেই, table আছে DELETE FROM users WHERE id = 5; -- শুধু id=5 মুছবে
⚡ Interview tip: TRUNCATE faster কারণ এটা DDL, transaction log লেখে না। DELETE DML, log লেখে তাই rollback করা যায়।
04 / Beginner
DML — INSERT, UPDATE, DELETE
Data Manipulation Language — data রাখা, পরিবর্তন এবং মুছার জন্য।
➕INSERT
-- একটি row INSERT INTO users (name, email, age) VALUES ('Rahim', 'rahim@gmail.com', 25); -- একসাথে অনেক row INSERT INTO users (name, email, age) VALUES ('Karim', 'karim@gmail.com', 30), ('Sumon', 'sumon@gmail.com', 22), ('Rima', 'rima@gmail.com', 28); -- অন্য table থেকে data নিয়ে insert INSERT INTO users_backup SELECT * FROM users WHERE age > 25;
✏️UPDATE
-- একটি row update UPDATE users SET age = 26, name = 'Rahim Ali' WHERE id = 1; -- অনেক row একসাথে UPDATE products SET price = price * 1.1 -- 10% price বাড়াও WHERE category_id = 3; -- CASE দিয়ে conditional update UPDATE orders SET status = CASE WHEN total > 5000 THEN 'premium' WHEN total > 1000 THEN 'regular' ELSE 'basic' END;
⚠️ সাবধান: UPDATE এ WHERE না দিলে সব row update হবে!
🗑️DELETE
-- একটি row DELETE FROM users WHERE id = 5; -- Condition দিয়ে DELETE FROM users WHERE age < 18; -- Subquery দিয়ে delete DELETE FROM orders WHERE user_id IN ( SELECT id FROM users WHERE status = 'banned' );
05 / Beginner
SELECT & Filtering
Data পড়ার জন্য SELECT। এটাই সবচেয়ে বেশি ব্যবহৃত SQL command।
🔍SELECT এর পূর্ণ Syntax
SELECT column1, column2 -- কোন column চাই FROM table_name -- কোন table থেকে WHERE condition -- filter GROUP BY column -- group করো HAVING group_condition -- group এর filter ORDER BY column ASC/DESC -- sort করো LIMIT n OFFSET m; -- কতটুকু নেবো
💡 Execution order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
🎯WHERE Conditions
-- Basic comparison SELECT * FROM users WHERE age > 25; SELECT * FROM users WHERE name = 'Rahim'; SELECT * FROM users WHERE age != 30; -- AND / OR / NOT SELECT * FROM users WHERE age > 20 AND age < 40; SELECT * FROM users WHERE city = 'Dhaka' OR city = 'Chittagong'; -- BETWEEN (inclusive) SELECT * FROM users WHERE age BETWEEN 20 AND 30; -- IN SELECT * FROM users WHERE city IN ('Dhaka', 'Chittagong', 'Sylhet'); -- LIKE (pattern matching) SELECT * FROM users WHERE name LIKE 'R%'; -- R দিয়ে শুরু SELECT * FROM users WHERE email LIKE '%gmail%'; -- gmail আছে SELECT * FROM users WHERE name LIKE '_ahim'; -- _ = যেকোনো ১ char -- NULL check SELECT * FROM users WHERE phone IS NULL; SELECT * FROM users WHERE phone IS NOT NULL; -- LIMIT & OFFSET (pagination) SELECT * FROM users ORDER BY id LIMIT 10 OFFSET 20; -- page 3
📋DISTINCT, ORDER BY, ALIAS
-- DISTINCT — duplicate বাদ SELECT DISTINCT city FROM users; -- ORDER BY SELECT * FROM users ORDER BY name ASC; SELECT * FROM users ORDER BY age DESC, name ASC; -- ALIAS (AS) SELECT name AS full_name, age AS years FROM users; SELECT u.name, o.total FROM users u, orders o; -- table alias -- String functions SELECT UPPER(name), LOWER(email), LENGTH(name) FROM users; SELECT CONCAT(first_name, ' ', last_name) AS full_name FROM users; SELECT SUBSTRING(email, 1, 5) FROM users;
06 / Intermediate
JOINs
একাধিক table কে একসাথে query করার জন্য JOIN। Database এর সবচেয়ে গুরুত্বপূর্ণ concept।
🔗JOIN এর ধরন — Visual
JOIN Typeকী পাবোNULL হয়?
INNER JOINদুই table এ match আছে এমন rowনা
LEFT JOINLeft table সব + Right এ matchRight এ NULL হতে পারে
RIGHT JOINRight table সব + Left এ matchLeft এ NULL হতে পারে
FULL OUTER JOINদুই table এর সব rowদুই দিকেই NULL হতে পারে
CROSS JOINCartesian productনা
SELF JOINTable নিজের সাথে join—
⚡INNER JOIN

দুই table এ common record — সবচেয়ে বেশি ব্যবহৃত।

-- Users এবং তাদের orders SELECT u.name, u.email, o.id AS order_id, o.total FROM users u INNER JOIN orders o ON u.id = o.user_id; -- Result: শুধু যেসব user এর order আছে
⬅️LEFT JOIN

সব user দেখাবে, order না থাকলে NULL।

SELECT u.name, o.id AS order_id, o.total FROM users u LEFT JOIN orders o ON u.id = o.user_id; -- Result: -- Rahim | 1 | 500 -- Karim | NULL | NULL ← কোনো order নেই -- Sumon | 2 | 1200 -- Order নেই এমন user খোঁজা SELECT u.name FROM users u LEFT JOIN orders o ON u.id = o.user_id WHERE o.id IS NULL;
🔄SELF JOIN

Employee table এ manager খোঁজা — একই table দুইবার।

-- employees table: id, name, manager_id SELECT e.name AS employee, m.name AS manager FROM employees e LEFT JOIN employees m ON e.manager_id = m.id;
🔀3 Table JOIN
-- users → orders → products SELECT u.name, p.name AS product, o.quantity, o.total FROM users u JOIN orders o ON u.id = o.user_id JOIN products p ON o.product_id = p.id WHERE u.id = 1;
💡 মনে রাখো: JOIN এর সংখ্যা = table সংখ্যা - 1
07 / Intermediate
Aggregate Functions
অনেক row এর উপর calculation করার জন্য — COUNT, SUM, AVG, MAX, MIN।
🧮সব Aggregate Functions
-- COUNT — কয়টা row SELECT COUNT(*) FROM users; -- সব row SELECT COUNT(phone) FROM users; -- NULL বাদে SELECT COUNT(DISTINCT city) FROM users; -- unique city -- SUM — যোগফল SELECT SUM(total) FROM orders; SELECT SUM(total) FROM orders WHERE status = 'paid'; -- AVG — গড় SELECT AVG(age) FROM users; SELECT ROUND(AVG(price), 2) FROM products; -- MAX / MIN SELECT MAX(price) FROM products; SELECT MIN(age), MAX(age) FROM users; -- একসাথে SELECT COUNT(*) AS total_orders, SUM(total) AS revenue, AVG(total) AS avg_order, MAX(total) AS biggest_order FROM orders;
08 / Intermediate
GROUP BY & HAVING
Data কে group করে aggregate calculation করার জন্য।
📊GROUP BY
-- প্রতি city তে কতজন user SELECT city, COUNT(*) AS total FROM users GROUP BY city; -- প্রতি category তে মোট sales SELECT c.name AS category, COUNT(p.id) AS products, SUM(o.total) AS total_sales FROM categories c JOIN products p ON c.id = p.category_id JOIN order_items o ON p.id = o.product_id GROUP BY c.id, c.name ORDER BY total_sales DESC;
🎯HAVING — Group এর Filter

WHERE group এর আগে কাজ করে, HAVING group এর পরে।

-- ৩ এর বেশি order আছে এমন user SELECT user_id, COUNT(*) AS order_count FROM orders GROUP BY user_id HAVING order_count > 3; -- মোট sales ১০০০০ এর বেশি এমন category SELECT category_id, SUM(total) AS revenue FROM orders GROUP BY category_id HAVING revenue > 10000 ORDER BY revenue DESC;
⚡ WHERE vs HAVING: WHERE aggregate function ব্যবহার করতে পারে না। HAVING পারে। WHERE সব সময় GROUP BY এর আগে আসে।
09 / Intermediate
Subquery
Query এর ভেতরে আরেকটি query। Complex problem সমাধানের শক্তিশালী হাতিয়ার।
🪆Subquery Types
── 1. WHERE এ Subquery -- সবচেয়ে বেশি দামি product SELECT name, price FROM products WHERE price = (SELECT MAX(price) FROM products); -- Average এর বেশি বয়স্ক user SELECT name, age FROM users WHERE age > (SELECT AVG(age) FROM users); ── 2. IN দিয়ে Subquery -- যেসব user order করেছে SELECT name FROM users WHERE id IN (SELECT DISTINCT user_id FROM orders); ── 3. FROM এ Subquery (Derived Table) SELECT * FROM ( SELECT user_id, SUM(total) AS spent FROM orders GROUP BY user_id ) AS user_spending WHERE spent > 5000; ── 4. EXISTS -- কমপক্ষে একটি order আছে এমন user SELECT name FROM users u WHERE EXISTS ( SELECT 1 FROM orders o WHERE o.user_id = u.id ); ── 5. Correlated Subquery (ধীরে কিন্তু powerful) -- প্রতি user এর সর্বোচ্চ order SELECT u.name, o.total FROM orders o JOIN users u ON o.user_id = u.id WHERE o.total = ( SELECT MAX(total) FROM orders WHERE user_id = o.user_id );
💡 Performance tip: Correlated subquery প্রতিটি row এর জন্য run হয়। বড় data তে JOIN বা CTE ব্যবহার করো।

11 / Intermediate
Normalization
Database design এর নিয়ম — redundancy কমাও, integrity বাড়াও।
📐Normal Forms

1NF — First Normal Form

প্রতিটি cell এ একটাই value, কোনো repeating group নেই।

❌ 1NF নয়
id | name | phones 1 | Rahim | 017,018 ← multiple!
✅ 1NF
id | name | phone 1 | Rahim | 017... 1 | Rahim | 018...

2NF — Second Normal Form

1NF + Partial dependency নেই — Non-key column কে সম্পূর্ণ Primary Key এর উপর depend করতে হবে।

💡 Composite PK থাকলে প্রাসঙ্গিক। PK শুধু একটি column হলে 2NF automatically satisfied।
❌ 2NF নয় (PK: order_id + product_id)
order_id | product_id | qty | product_name | ← product_name শুধু product_id এর উপর depend করে
✅ 2NF — আলাদা করো
-- order_items table order_id | product_id | qty -- products table product_id | product_name

3NF — Third Normal Form

2NF + Transitive dependency নেই — Non-key column অন্য non-key column এর উপর depend করতে পারবে না।

❌ 3NF নয়
id | name | zip | city city depends on zip, not on id
✅ 3NF
-- users: id, name, zip -- zip_codes: zip, city
⚖️Normalization vs Denormalization
—NormalizationDenormalization
Redundancyকমবেশি
Write speedভালোধীর
Read speedJOIN লাগে, একটু ধীরদ্রুত
কখন ব্যবহারOLTP (transactions)OLAP (analytics, reporting)
12 / Advanced
Indexing
Query কে rocket fast করার জন্য। Book এর index এর মতো — পুরো book না পড়েও page খুঁজে পাবে।
⚡Index তৈরি
-- Simple index CREATE INDEX idx_email ON users(email); -- Unique index CREATE UNIQUE INDEX idx_unique_email ON users(email); -- Composite index (column এর order গুরুত্বপূর্ণ) CREATE INDEX idx_name_city ON users(name, city); -- Index দেখা SHOW INDEX FROM users; -- Index মুছা DROP INDEX idx_email ON users; -- EXPLAIN দিয়ে index ব্যবহার হচ্ছে কিনা দেখো EXPLAIN SELECT * FROM users WHERE email = 'test@test.com';
📋Index কখন দেবো, কখন দেবো না
✅ Index দাও যখন:
• WHERE তে বারবার ব্যবহার হয়
• JOIN এ foreign key column
• ORDER BY, GROUP BY column
• UNIQUE দরকার
• Read-heavy table
❌ Index দিও না যখন:
• Table ছোট (< ১০০০ row)
• Write-heavy table (INSERT/UPDATE slow হয়)
• Low cardinality column (gender, status)
• Rarely searched column
🌳Index এর ধরন
TypeDescription
B-TreeDefault। Range queries, sorting এর জন্য ভালো।
HashExact match এর জন্য দ্রুত। Range query support করে না।
Full-TextText search এর জন্য (LIKE '%word%' slow — Full-Text fast)
SpatialGeographic data (GPS coordinates)
ClusteredData physically sorted করে রাখে (Primary Key)
Non-ClusteredSeparate structure, pointer দিয়ে data খোঁজে
13 / Advanced
Views & CTE
Complex query কে reusable এবং readable করার জন্য।
👁️VIEW — Virtual Table

একটি saved query — বারবার লিখতে হবে না, security layer হিসেবেও কাজ করে।

-- View তৈরি CREATE VIEW active_users AS SELECT id, name, email, created_at FROM users WHERE status = 'active'; -- View ব্যবহার (table এর মতো) SELECT * FROM active_users; -- Complex view CREATE VIEW user_order_summary AS SELECT u.id, u.name, COUNT(o.id) AS total_orders, SUM(o.total) AS total_spent FROM users u LEFT JOIN orders o ON u.id = o.user_id GROUP BY u.id, u.name; -- View update/delete CREATE OR REPLACE VIEW active_users AS ...; DROP VIEW active_users;
🔗CTE — Common Table Expression

Query এর মধ্যে temporary named result। View এর মতো কিন্তু শুধু ঐ query তেই কাজ করে।

-- Basic CTE WITH high_spenders AS ( SELECT user_id, SUM(total) AS spent FROM orders GROUP BY user_id HAVING spent > 10000 ) SELECT u.name, hs.spent FROM users u JOIN high_spenders hs ON u.id = hs.user_id; -- Multiple CTEs WITH monthly_sales AS ( SELECT MONTH(created_at) AS month, SUM(total) AS sales FROM orders GROUP BY month ), avg_sales AS ( SELECT AVG(sales) AS avg FROM monthly_sales ) SELECT m.month, m.sales, a.avg FROM monthly_sales m, avg_sales a WHERE m.sales > a.avg; -- Recursive CTE (hierarchy) WITH RECURSIVE emp_hierarchy AS ( -- Base case: top manager SELECT id, name, manager_id, 0 AS level FROM employees WHERE manager_id IS NULL UNION ALL -- Recursive case SELECT e.id, e.name, e.manager_id, eh.level + 1 FROM employees e JOIN emp_hierarchy eh ON e.manager_id = eh.id ) SELECT * FROM emp_hierarchy;
14 / Advanced
Transactions & ACID
অনেকগুলো operation কে একটি unit হিসেবে execute করা — হয় সব হবে, না হয় কিছুই না।
🧪ACID Properties
Propertyমানে কীউদাহরণ
Atomicityসব হবে অথবা কিছুই নাBank transfer — টাকা কাটলে অবশ্যই জমা হবে
ConsistencyData সবসময় valid state এ থাকবেBalance কখনো negative হবে না
Isolationএকটি transaction অন্যটিকে affect করবে নাদুই জন একই account এ টাকা তুললেও ঠিক থাকবে
DurabilityCommit হলে permanently savedPower চলে গেলেও data থাকবে
💳Transaction Example
-- Bank transfer: A থেকে B তে 1000 টাকা START TRANSACTION; UPDATE accounts SET balance = balance - 1000 WHERE id = 1; -- A এর balance কমাও UPDATE accounts SET balance = balance + 1000 WHERE id = 2; -- B তে যোগ করো -- সব ঠিক থাকলে COMMIT; -- যেকোনো error হলে ROLLBACK; -- SAVEPOINT START TRANSACTION; INSERT INTO orders ... ; SAVEPOINT order_placed; INSERT INTO payments ... ; ROLLBACK TO SAVEPOINT order_placed; -- শুধু payment rollback COMMIT;

Window Functions
Row collapse না করে aggregate calculation করে — GROUP BY এর চেয়ে বেশি powerful।
🪟Window Function Syntax
function_name() OVER ( PARTITION BY column -- group তৈরি ORDER BY column -- sort ROWS BETWEEN ... -- frame define )
📈Ranking Functions
-- ROW_NUMBER: unique sequential number SELECT name, salary, ROW_NUMBER() OVER (ORDER BY salary DESC) AS rank FROM employees; -- RANK: same value → same rank, gap তৈরি হয় -- DENSE_RANK: same value → same rank, gap নেই SELECT name, score, RANK() OVER (ORDER BY score DESC) AS rank, DENSE_RANK() OVER (ORDER BY score DESC) AS dense_rank FROM students; -- Score: 90, 90, 80 -- RANK: 1, 1, 3 ← gap আছে -- DENSE_RANK:1, 1, 2 ← gap নেই -- NTILE: n ভাগে ভাগ করো SELECT name, salary, NTILE(4) OVER (ORDER BY salary) AS quartile FROM employees;
🧮Analytic Functions
-- LAG / LEAD: আগের/পরের row এর value SELECT month, sales, LAG(sales) OVER (ORDER BY month) AS prev_month, sales - LAG(sales) OVER (ORDER BY month) AS growth FROM monthly_sales; -- SUM OVER PARTITION: group এর মধ্যে running total SELECT department, name, salary, SUM(salary) OVER (PARTITION BY department) AS dept_total, AVG(salary) OVER (PARTITION BY department) AS dept_avg FROM employees; -- Running total SELECT created_at, total, SUM(total) OVER (ORDER BY created_at) AS running_total FROM orders; -- প্রতি department এ top earner SELECT * FROM ( SELECT *, ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS rn FROM employees ) t WHERE rn = 1;

🍃CAP Theorem

Distributed system এ এই তিনটির যেকোনো দুটি পাওয়া যায়, তিনটি একসাথে নয়:

C — Consistency: সব node একই data দেখায়
A — Availability: সবসময় response পাবো
P — Partition Tolerance: Network ভেঙে গেলেও কাজ করে
MySQL/PostgreSQL: CA (Partition সহ্য করতে পারে না)
MongoDB: CP (Consistency + Partition)
Cassandra: AP (Availability + Partition)

⚡ Interview Q&A
সবচেয়ে বেশি জিজ্ঞেস করা প্রশ্ন এবং perfect উত্তর।
Q1. DELETE vs TRUNCATE vs DROP এর পার্থক্য?
উত্তর: DELETE DML — condition দিয়ে specific row মোছে, transaction log রাখে তাই rollback সম্ভব। TRUNCATE DDL — সব row মোছে কিন্তু structure রাখে, log রাখে না তাই দ্রুত কিন্তু rollback হয় না। DROP DDL — পুরো table structure সহ মুছে ফেলে।
Q2. INNER JOIN vs LEFT JOIN এর পার্থক্য?
উত্তর: INNER JOIN শুধু দুই table এ match আছে এমন row দেয়। LEFT JOIN বাম table এর সব row দেয়, ডান table এ match না থাকলে NULL আসে। উদাহরণ: সব user দেখাতে চাইলে LEFT JOIN, শুধু order করা user দেখাতে চাইলে INNER JOIN।
Q3. Index কী এবং কখন ব্যবহার করবো?
উত্তর: Index হলো database এর lookup table যা data খোঁজা দ্রুত করে। B-Tree structure ব্যবহার করে। WHERE, JOIN, ORDER BY column এ index দিলে read fast হয়। কিন্তু index INSERT/UPDATE slow করে কারণ index ও update করতে হয়। Low cardinality column (gender) এ index অকেজো।
Q4. ACID কী? প্রতিটি explain করো।
উত্তর: Atomicity — transaction এর সব operation হবে নয়তো কিছুই না। Consistency — transaction শেষে data valid state এ থাকবে। Isolation — concurrent transaction গুলো একে অপরকে affect করবে না। Durability — commit হলে data permanently saved, system failure এও টিকবে।
Q5. Normalization এর লক্ষ্য কী? 1NF, 2NF, 3NF explain করো।
উত্তর: Redundancy কমানো এবং data integrity বাড়ানো। 1NF: প্রতি cell এ একটি value, কোনো repeating group নয়। 2NF: 1NF + Partial dependency নেই (সব non-key column সম্পূর্ণ PK এর উপর depend করে)। 3NF: 2NF + Transitive dependency নেই (non-key column অন্য non-key column এর উপর depend করে না)।
Q6. WHERE vs HAVING পার্থক্য?
উত্তর: WHERE GROUP BY এর আগে কাজ করে, individual row filter করে, aggregate function ব্যবহার করতে পারে না। HAVING GROUP BY এর পরে কাজ করে, grouped result filter করে, aggregate function ব্যবহার করতে পারে। যেমন: "১০ এর বেশি order আছে এমন user" — এটা HAVING দিয়ে করতে হবে।
Q7. Window Function vs GROUP BY পার্থক্য?
উত্তর: GROUP BY row collapse করে — ফলে original row এর detail হারিয়ে যায়। Window Function row collapse করে না — প্রতিটি row থাকে কিন্তু সাথে aggregate value যোগ হয়। যেমন প্রতিটি employee এর salary এবং সাথে department এর average salary দেখাতে Window Function লাগবে।
Q8. View কী? Stored Procedure এর সাথে পার্থক্য?
উত্তর: View হলো stored SELECT query — virtual table, data store করে না। Stored Procedure হলো stored executable code — parameter নিতে পারে, DML করতে পারে, logic লেখা যায়। View শুধু SELECT, Procedure INSERT/UPDATE/DELETE সব করতে পারে।
